# 代码库镜像说明

## 0. 文档定位

本文档是对当前仓库现有代码的镜像映射，而不是对理想架构的描述。它忠实记录当前代码里每一个实际存在的文件、类、函数、关键变量、输入输出和数据处理过程，并且尽量说明它们与 [AGENTS.md](AGENTS.md) 以及 [docs/](docs/) 文档体系之间的对应关系。

修改纪律：如果代码发生变化，必须先修改本文档，再修改代码。这样才能保证本文档始终是工程文件的镜像，而不是过期注释。

覆盖范围：
- 逐文件、逐函数说明仓库自有代码。
- 说明当前主流程实际调用到的 TWMM 上游代码。
- 对第三方运行时附带目录做边界说明，不把它们伪装成项目自有源码。

明确边界：`Metashape/App/Metashape/python/` 下的大量 CPython 标准库与打包依赖属于运行时附带代码，不是本项目自己写的业务源码，本文不逐文件展开其内部实现；只说明它作为运行环境的角色。

## 2.4 运行日志体系

这次改动的目标不是改变算法流程，而是把运行期行为变成可追踪、可复盘、可审计的日志轨迹。

当前实现已经新增统一日志入口 [src/pipeline_io/logging_setup.py](src/pipeline_io/logging_setup.py)：
- 在 `run_dir/logs/run.log` 下写入文件日志，保留完整运行过程。
- 控制台继续输出阶段摘要，便于交互式运行时观察进度。
- 日志格式统一为时间戳、级别、logger 名称和消息正文。
- `main.py` 会在阶段开始、阶段结束、异常失败时记录结构化信息，并带上耗时和结果摘要。

这意味着：
- 原有算法输出和数据契约没有因为日志改造而改变。
- 变更影响的是运行轨迹可见性，而不是 TWMM、辐射校正或点云重建的数值流程。
- `src/metashape_reconstruction/photogrammetry.py` 的日志改动只是把模块内 `basicConfig` 改成普通 logger，以便接入统一日志系统；它不改变 Metashape 调用的 SfM/MVS 计算步骤。

## 3. 逐文件镜像说明

### 3.1 [main.py](main.py)

**依据**：AGENTS.md 的“配置、数据入口层与主引擎”；docs/architecture.md 的运行模式与 config 层约束；docs/runtime_config.md 的运行参数契约。

**模块作用**：
- 这是当前仓库真正的命令行入口文件。
- 它负责接收命令行参数，构造参数对象，然后调用具体层的实现。
- 它不是算法层，不做去畸变、匹配、辐射或重建本体计算。

**模块级导入**：
- `argparse`：构建 CLI。
- `Path`：让路径参数在 CLI 层直接变成路径对象。
- `EnvironmentParameters`, `RadiometryParams`, `TwmmParams`：来自 [src/config/runtime_models.py](src/config/runtime_models.py) 的参数结构。
- `ConfigManager`：来自 [src/config/config_manager.py](src/config/config_manager.py) 的运行上下文管理器。
- `TwmmAdapter`：来自 [src/matching/twmm_adapter.py](src/matching/twmm_adapter.py) 的跨光谱匹配适配器。
- `RadiometryPipeline`：来自 [src/radiometry/temperature_pipeline.py](src/radiometry/temperature_pipeline.py) 的热辐射管道。

**函数级说明**：

| 函数 | 输入 | 输出 | 说明 |
|---|---|---|---|
| `_build_parser()` | 无 | `argparse.ArgumentParser` | 仅保留 `run_all` 入口；支持可选兼容参数 `command=run_all` 与 `--config-file` |
| `run_all(args)` | `--config-file` 参数 | `None`，执行 1-9 层编排 | 读取 `ConfigManager`、执行标定/去畸变/匹配/辐射/重建/重投影/富集/导出，并更新运行清单 |
| `main()` | 无 | `None` | 解析参数后直接调用 `run_all(args)` |

**关键变量与真实含义**：
- `args.command`：仅兼容保留，允许值只有 `run_all`；可省略。
- `args.config_file`：配置来源，既可传 JSON 文件，也可传工作区路径。
- `manager.user_config`：`run_all` 在内部按阶段读取 TWMM、辐射、重建等参数，不再通过独立子命令参数传入。

**输入输出过程**：
- `run_all` 现在会真正串联 1-9 层。
- `run_all` 内部会按配置驱动 TWMM 与辐射处理，不再暴露独立 `matching/radiometry` 命令。

**当前代码的忠实备注**：
- `main.py` 已经是全局入口，也是当前的一键全链路编排器。
- 如果你把它当成 C 语言那种 `main()`，它现在同时承担入口存在与全链路执行两项职责。

### 3.2 [src/config/runtime_models.py](src/config/runtime_models.py)

**依据**：AGENTS.md 的配置入口层；docs/runtime_config.md；docs/matching_algorithm.md；docs/radiometry_model.md。

**模块作用**：
- 这里是强类型参数契约层。
- 它不做业务流程，只定义参数该长什么样、怎么检查合法性。

**模块级对象**：
- `ConfigError`：运行时配置校验失败时抛出的异常。
- `TwmmParams`：TWMM 匹配参数。
- `EnvironmentParameters`：辐射校正环境参数。
- `RadiometryParams`：辐射处理参数。
- `RuntimeConfig`：Path 版本的运行配置对象。
- `require_existing_paths()`：路径存在性检查。
- `parse_environment_parameters()`：从映射结构里解析环境参数。

**`TwmmParams` 字段**：
- `patch_size`：原子块边长。
- `search_radius`：搜索半径。
- `level_max`：金字塔最大层数。
- `crop_size`：裁剪尺寸，`Tuple[int, int]`。
- `crop_offset`：裁剪偏移，默认 `(0, 0)`。
- `scale`：双光谱缩放字典，必须同时包含 `thermal` 和 `visible`。
- `outlier_threshold_px`：外点阈值。
- `min_inliers`：最少内点数。
- `homography_condition_max`：单应性条件数上界。

**`TwmmParams.validate()` 的真实行为**：
- 检查 `patch_size > 0`。
- 检查 `search_radius > 0`。
- 检查 `level_max > 0`。
- 检查 `crop_size` 是两个正整数。
- 检查 `scale` 是 dict，且含 `thermal` 与 `visible`。
- 不检查图像文件是否存在，也不检查命名匹配。

**`EnvironmentParameters` 字段**：
- `ambient_temperature_celsius`
- `relative_humidity_percent`
- `emissivity_ratio`
- `distance_to_target_m`
- `reflected_temperature_celsius`
- `atmospheric_pressure_hpa`
- `source`
- `source_ref`

**`EnvironmentParameters.missing_required()` 的真实行为**：
- 只要前四个字段有任意一个是 `None`，就会被判为缺失。
- 默认值在类定义里已经给了 `-1.0 / 70.0 / 0.97 / 100.0`，所以如果直接实例化而不覆写，这个方法会返回空列表。
- 当前主入口是 `run_all`，环境参数由配置体系解析后传入辐射阶段，不通过独立 `run_radiometry` CLI 参数注入。

**`RadiometryParams` 字段**：
- `metadata_json_path`
- `processing_version`
- `temperature_unit`
- `input_is_temperature`
- `raw_to_temperature_scale`
- `raw_to_temperature_offset`
- `valid_range_celsius`

**`RuntimeConfig` 字段**：
- `workspace_root`
- `output_root`
- `dataset_id`
- `run_id`
- `pipeline_version`
- `stage_selection`
- `environment_parameters`
- `twmm_params`
- `radiometry_params`
- `quality_thresholds`
- `toolchain_versions`

**`RuntimeConfig.validate_stage_selection()` 的真实行为**：
- 允许的阶段是 `config`、`calibration`、`preprocess`、`matching`、`radiometry`、`metashape`、`reconstruction`、`geometry`、`enrichment`、`io`、`validation`。
- `reconstruction` 作为旧名别名继续保留，实际运行清单会归一化为 `metashape`。

**辅助函数**：
- `require_existing_paths(paths, label)`：逐个检查 `Path.exists()`，缺失就抛 `ConfigError`。
- `parse_environment_parameters(raw)`：把 mapping 里的键读出来，缺失时填默认值 `-1.0 / 70.0 / 0.97 / 100.0`。

**当前代码的忠实备注**：
- 这个文件里的 `RuntimeConfig` 和 [src/config/config_manager.py](src/config/config_manager.py) 里的 `RuntimeConfig` 不是同一个类，字段类型也不一样。
- 这是当前代码的真实情况，镜像文档里必须明确区分，不能混写成一个概念。

### 3.3 [src/config/config_manager.py](src/config/config_manager.py)

**依据**：AGENTS.md 的配置与数据入口层；docs/runtime_config.md；docs/dataset_profile.md。

**模块作用**：
- 这是运行时上下文管理器。
- 它负责把用户传入的配置、默认值和目录结构组装成一次运行的工作空间。
- 它不做算法，只做路径、清单和目录准备。

**模块级对象**：
- `DatasetProfile`：默认物理事实与别名。
- `RuntimeConfig`：这个文件里自己的字符串版运行配置结构。
- `ConfigManager`：运行上下文管理主类。

**`DatasetProfile` 字段**：
- `dataset_id = "test_Arctic"`
- `rgb_resolution = (4032, 3024)`
- `thermal_resolution = (1280, 1024)`
- `chessboard_physical = (12, 9)`
- `chessboard_size = (11, 8)`
- `chessboard_corners = (12, 9)`
- `rgb_camera_matrix = None`
- `thermal_camera_matrix = None`
- `square_size_mm = 40.0`

**`RuntimeConfig` 字段**：
- `workspace_root`
- `dataset_id`
- `input_dataset_root`
- `calibration_dataset_root`
- `output_root`
- `run_id`
- `pipeline_version`
- `stage_selection`
- `quality_thresholds`
- `environment_parameters`
- `toolchain_versions`
- `data_dir_rgb`
- `data_dir_thermal_tiff`

**`ConfigManager.__init__()` 的真实行为**：
- 先建一个空 `cfg`。
- 把 `kwargs` 先合并进去。
- 如果 `user_config` 是字符串且是文件，就按 JSON 读入并合并。
- 如果 `user_config` 是字符串但不是文件，就把它当成 `workspace_root`。
- 如果 `user_config` 是 dict，就合并进 `cfg`。
- 否则抛 `ValueError`。
- 然后创建 `dataset_profile`。
- 接着调用 `_build_runtime_config(cfg)`。
- 再把 `self.config = self.runtime_config` 作为兼容别名。
- 最后设置 `workspace_root`、`run_id`、`output_root`、`run_dir`，并调用 `setup_workspace()`。

**`_build_runtime_config(user_cfg)` 的真实行为**：
- 用 UTC 时间戳和 JSON 字符串的 SHA256 前 8 位生成 `run_id`。
- `dataset_id` 默认使用 `DatasetProfile.dataset_id`。
- `workspace_root` 默认是 `os.getcwd()`。
- `output_root` 默认是 `workspace_root/runs`。
- `input_dataset_root` 默认是 `workspace_root/test_Arctic`。
- `calibration_dataset_root` 默认是 `workspace_root/M400-H30T-CALIB-CHESSBOARD`。
- `data_dir_rgb` 默认 `input_dataset_root/rgb_dir`。
- `data_dir_thermal_tiff` 默认 `input_dataset_root/tiff_dir`。
- `stage_selection` 默认是 `['config', 'calibration', 'preprocess', 'matching', 'radiometry', 'metashape', 'geometry', 'enrichment', 'validation']`。
- `quality_thresholds` 默认空 dict。
- `environment_parameters` 直接保留 `user_cfg` 中的 dict，不做强类型转换。
- `toolchain_versions` 默认记录当前 Python 版本、Metashape 2.2.1、TWMM-main/TAMM_clean、ExifTool 13.48 和 pipeline schema 1.0.0。

**`setup_workspace()` 的真实行为**：
- 创建 `run_dir`。
- 创建这些子目录：`manifest`、`calibration`、`undistorted`、`preprocess`、`matching`、`radiometry`、`metashape`、`geometry`、`enrichment`、`validation`、`reports`。
- 然后调用 `_generate_run_manifest()`。

**`_generate_run_manifest()` 的真实行为**：
- 写出 `manifest/run_manifest.json`。
- 里面包含 `run_id`、`dataset_id`、`created_at`、`pipeline_version`、`input_paths`、`output_paths`、`stage_status`、`runtime_config_snapshot`。
- `stage_status` 初始都是 `pending`。
- 失败时只记录 warning，不抛异常。

**其他方法**：
- `_resolve_path(path_like)`：空字符串返回 workspace_root，绝对路径原样返回，相对路径拼到 workspace_root。
- `get_stage_dir(stage_name)`：返回 `dirs` 里已有目录，或者 `run_dir/stage_name`。
- `get_calibration_output_path()`：返回 `calibration/sensors_calibration_<run_id>.json`。

**当前代码的忠实备注**：
- 这里会在构建时归一化 `reconstruction -> metashape`，因此不会再把旧名传播到运行清单里。
- `environment_parameters` 在这个类里只是原始 dict 快照，不是 [src/config/runtime_models.py](src/config/runtime_models.py) 里的 `EnvironmentParameters`。

### 3.5 [src/pipeline_io/io_utils.py](src/pipeline_io/io_utils.py)

**依据**：docs/file_formats.md；AGENTS.md 的 I/O 支撑边界。

**模块作用**：
- 提供路径敏感读写的兜底函数。
- 主要是为了 Windows 下含中文路径、OpenCV 不稳定读写等问题做 fallback。

**函数**：
- `safe_imread(path, flags=cv2.IMREAD_COLOR)`：先用 `cv2.imread`，失败后用 `open(..., 'rb')` + `np.frombuffer` + `cv2.imdecode`。
- `safe_imwrite(path, img)`：先用 `cv2.imwrite`，失败后用 `cv2.imencode` 再手动写文件。

**当前状态**：
- `safe_imread` 被 [src/calibration/calibrator.py](src/calibration/calibrator.py) 和 [src/preprocess/undistort.py](src/preprocess/undistort.py) 使用。
- `safe_imwrite` 当前在核心代码里基本没有被直接调用，但它确实存在。

- 只有第一次调用时才执行。
- 先检查 `twmm_root/TAMM_clean` 是否存在。
- 把 `TAMM_clean` 目录加入 `sys.path`。
- 导入 `thermal_visible` 和 `image_process`。
- 缓存 `ThermalVisble`、`filter_outliers`、`findHomoraphy`。

**`match_pair()` 的真实处理流程**：
1. 校验 `TwmmParams`。
2. 检查 RGB 与热红外 TIFF 文件存在。
3. 预览图不存在时，回退到热红外 TIFF 本身。
4. 创建输出目录 `output_dir/pair_id`。
5. 实例化 `thermal_visible.ThermalVisble`，传入：
   - `thermal_tiff_path`
   - `thermal_rgb_path`（预览图或 TIFF 回退）
   - `visible_rgb_path`
   - `crop_size=params.crop_size`
   - `scale=params.scale`
   - `attention_flag=False`
   - `crop_offset=params.crop_offset`
6. 调 `get_img_features(method="CFOG", bin_size=9)`。
7. 调 `get_correspoints(method="TAMM", patch_size=..., search_radius=..., level_max=...)`。
8. 把返回的热点/可见点字典拆成列表。
9. 调 TWMM 的 `filter_outliers(..., thresh=params.outlier_threshold_px, method="NBCS")`。
10. 组装 `correspondences` 列表。
11. 组装 `good` 对并调 `findHomoraphy(good)`。
12. 计算内点数、外点数、条件数、置信度、质量标签。
13. 写 `match_overlay.png`、`correspondences.csv`、`homography_matrix.csv`、`match_result.json`。
14. 返回 `MatchResult`。

**`match_pair()` 的当前约束与真实细节**：
- `get_img_features` 固定用 CFOG，`bin_size=9`，因为上游 CFOG 代码里有硬编码 9 的逻辑。
- `findHomoraphy` 的返回值在上游有潜在不稳定情况，当前适配器按返回值直接判断。
- `diagnostics.failure_reason` 现在只写一个通用字符串，不会细分具体失败原因。
- `homography_matrix.csv` 的写法是按矩阵三行写文件，header 仍然是 `h11,h12,h13`，这只是当前实现方式，不是理想格式。

**`match_batch_from_dirs()` 的真实行为**：
- 只扫描 `*.tif` 和 `*.tiff`。
- 按热红外文件名派生 RGB 文件名和预览图文件名。

**`export_reprojections()` 的真实行为**：

### 3.16 [src/enrichment/thermal_enrichment.py](src/enrichment/thermal_enrichment.py)

**依据**：AGENTS.md 第 8 层；docs/reconstruction_and_enrichment.md；docs/file_formats.md。

**模块作用**：
- 把热红外温度通过单应性映射赋给三维点。
- 当前也是独立实现，并已接入 `main.py` 的 `run_all`。

**类：`ThermalEnricher`**

| 成员 | 说明 |
|---|---|
| `__init__(homography, thermal_data, config)` | 保存单应性、热图字典和配置 |
| `bilinear_interpolate(img, u, v)` | 双线性插值，遇到 NaN 则返回 NaN |
| `enrich_point_cloud(point_cloud, reprojection_data)` | 对点云逐点计算温度、支持视角数和融合权重 |

**`__init__` 的真实行为**：
- `thermal_resolution` 从 `config['thermal_extraction']['thermal_resolution']` 读，默认字符串是 `1280x1024`。
- 拆成 `thermal_w` 和 `thermal_h`。
- `epsilon = 1e-4`。

**`enrich_point_cloud()` 的真实行为**：
- 创建三个数组：
  - `temperatures`，初始全 `NaN`
  - `support_views`，初始全 0
  - `fusion_weights`，初始全 0
- 对每个点查 `reprojection_data[point_id]`。
- 对每个观测：
  - 用 `cam_id.replace("RGB", "NIR")` 作为热图 key 的回退映射。
  - 把 RGB 坐标乘单应性映射到热图坐标。
  - 双线性插值温度。
  - 权重按距离光学中心的倒数计算。
- 如果有至少一个有效观测，就写入温度、支持视角数、融合权重。
- 最后把结果挂到 `point_cloud.temperature`、`point_cloud.support_views`、`point_cloud.fusion_weights`。

**当前状态备注**：
- 这层强依赖上游命名习惯，尤其是 `RGB` -> `NIR` 的字符串替换逻辑。

| 成员 | 输入 | 输出/效果 | 说明 |

**`get_img_features()` 的真实行为**：

  - 返回热特征和可见光特征。

**`get_correspoints()` 的真实行为**：
- `method='TAMM'` 时：

  - 调 `get_level0_dict()`。
  - 调 `get_all_level_dict(level_max)`。

  - 调 `get_corres_points()`。

- `method='CFOG'` 时：
  - 调 `cfog_matching_points(...)`。
  - 返回对应点字典。

- 根据 `save_dict` 保存不同文件：
  - `visible_rgb`
  - `thermal_rgb`
  - `warp_thermal`
  - `mosaic`
  - `matchpoints_img`
  - `homo`
  - `corres_points`
  - `attention_thermal`
- `corres_points` 会计算重投影误差并写 CSV。

**当前状态备注**：
- 这是 `TwmmAdapter` 的上游容器类。
- 它本身就保留了多种实验路径，但当前项目只固定用 CFOG + TAMM。

### 4.2 [TWMM-main/TAMM_clean/image_process.py](TWMM-main/TAMM_clean/image_process.py)

**文件作用**：
- 这是 TWMM 上游的通用图像和几何工具集。
- `TwmmAdapter` 直接依赖它里面的 `filter_outliers` 和 `findHomoraphy`。

**模块级函数**：
- `fusion_image(thermal_img, rgb_img)`：网格状融合图。
- `stitch_img(imgList, hor_flag=1, outpath=None, spacing=10)`：水平或垂直拼图。
- `img_read(img_path, scale)`：读取 mat / tif / jpg / png，并支持缩放。
- `center_crop(img, crop_size=1000)`：中心裁剪。
- `bottom_right_crop(img, crop_size=1000, offset=(0, 0))`：从右下角裁剪并支持偏移。
- `homo_save(csv_path, H)`：把 3x3 单应性矩阵写成文本。
- `homo_parse(homo_path)`：读回单应性矩阵。
- `img_split(img, patch_size, overlap=0)`：切 patch。
- `SSD_template_matching(template, image, attention, return_score_flag)`：SSD 模板匹配。
- `show_map_and_img(map_img, norm_flag, color_flag=True, cv_show_name=None, outpath=None)`：把 map 归一化并可视化。
- `get_cores_point(pt_x, pt_y, homo)`：用单应性把一个点映射到另一平面。
- `filter_outliers(thermal_point_list, sen_point_list, thresh, method)`：外点剔除。
- `findHomoraphy(good)`：求单应性矩阵。
- `mosaic_img(img_A, img_B, patch_size=100)`：棋盘式/镶嵌式融合。

**关键函数的真实行为**：
- `img_read()`：
  - `.mat` 时读 `img1_scb`。
  - `.tif/.tiff` 时用 `rasterio` 读。
  - 其它格式用 `cv2.imread`。
  - 读取失败就抛 `RuntimeError`。
- `filter_outliers()`：
  - `method='NBCS'` 时用 `cv2.findHomography(..., cv2.RANSAC, 5.0)` 反复删外点。
  - 返回内点和外点列表。
- `findHomoraphy()`：
  - `len(good) > 4` 时调用 `cv2.findHomography(..., 0)`。
  - 少于等于 4 时会走 identity 逻辑，但当前代码里 `flag` 有潜在未定义风险，这是真实代码状态。

**当前状态备注**：
- `TwmmAdapter.match_pair()` 正是依赖 `filter_outliers` 和 `findHomoraphy`。
- 这两个函数的行为必须和适配器一起理解。

### 4.3 [TWMM-main/TAMM_clean/CFOG.py](TWMM-main/TAMM_clean/CFOG.py)

**文件作用**：
- CFOG 特征和基于 Harris/SSD 的初始匹配实现。
- 适配器当前没有直接走这个分支，但 `ThermalVisble.get_img_features(method='CFOG')` 会用到它。

**模块级函数和类**：
- `keypoint_obtain(img, num_points, harris_thresh)`：分块 Harris 找关键点。
- `one_keypoint_matching(keypoint, fea_ref, fea_sen, attention, window_size=100, search_rad=60)`：单关键点模板匹配。
- `cfog_matching_points(ref_img, ref_fea, sen_fea, attention, window_size, search_rad, num_points)`：批量匹配，返回对应点字典。

**当前代码的忠实备注**：
