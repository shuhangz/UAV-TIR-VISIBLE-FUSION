# architecture.md

## 1. 文档目的

本文档定义本项目的端到端架构：从双光谱标定、去畸变、TWMM 跨光谱匹配、热红外辐射校正、Metashape 三维重建，到重投影、可见性判断、热信息富集和结果导出。

本文档的作用是让 AI 和人类维护者在任何新会话中都能快速恢复系统全貌，并明确各层的职责边界、输入输出和依赖方向。

## 2. 设计原则

- 先做几何一致性，再做热信息融合。
- 先保留原始数据，再生成派生数据。
- 先建立稳定的数据契约，再扩展实现细节。
- 热红外 TIFF 是匹配、辐射校正与热富集的主输入，JPG 仅用于预览、对照和诊断，不进入生产主链路。
- 生产主链路优先连接现有成熟实现，不重新发明 TWMM 或 Metashape 的核心算法。
- 热红外链路与可见光链路在匹配和富集层之前保持解耦。
- 所有阶段都必须可独立运行，并且输出可被下一阶段直接消费。

## 3. 端到端流程

本项目通过 `main.py` 作为如同 C 语言 `main()` 函数的全局编排入口，实现对各个阶段的可插拔调度。详见下表：

| 阶段 | 输入 | 输出 | 说明 |
|---|---|---|---|
| 0. 全局编排 (`main.py`) | 命令行参数或 JSON 配置 | 驱动各层流转的调用结果 | 结合 ConfigManager 实现1~9步的全链路串联 |
| 1. 配置与数据入口 | 数据集路径、相机参数、环境参数、运行模式 | 运行清单、RuntimeConfig、DatasetProfile | 统一管理路径、命名和日志 |
| 2. 双光谱标定 | 棋盘格影像、初始相机参数 | 内参、畸变参数、质量标记 | 生成后续去畸变和几何计算的基础 |
| 3. 图像去畸变 | 原始 RGB / 热红外 TIFF 图像、标定结果 | 几何校正后的图像 | 只做几何处理，不做辐射处理 |
| 4. 跨光谱匹配 | 去畸变后的 RGB / 热红外 TIFF 图像对 | 对应点、异常值剔除结果、单应性矩阵 | 以 TWMM 为核心 |
| 5. 热辐射校正 | 热红外 TIFF 主输入帧、元数据、环境参数 | 温度矩阵、单位、质量标记 | 独立于几何重建 |
| 6. 摄影测量重建 | RGB 图像、标定结果、项目配置 | 稠密点云、相机位姿、重投影记录 | 使用 Metashape 完成 SfM / MVS |
| 7. 重投影与可见性 | 稠密点云、相机位姿 | 点-图像对应、可见性、遮挡状态 | 只输出几何证据 |
| 8. 点云热富集 | 重投影记录、单应性、温度矩阵 | 带温度属性的三维点 | 采用插值和多视图融合 |
| 9. 导出与质检 | 各阶段结果 | PLY / LAS / CSV / JSON / 图像报告 | 供人工审核和回归验证 |

## 4. 模块边界

### 4.1 config

负责管理数据集路径、相机参数、环境参数、输出目录、运行模式和全局开关。config 不应承载算法逻辑，只负责读取 [docs/dataset_profile.md](dataset_profile.md) 的物理事实、读取 [docs/runtime_config.md](runtime_config.md) 的运行参数，并构造经过校验的运行配置对象、运行清单和阶段阈值快照。

### 4.2 calibration

负责棋盘格检测、相机标定、畸变建模、标定结果持久化和标定质量评估。该层只处理双光谱相机几何，不处理热辐射或点云富集。

### 4.3 preprocess

负责 RGB 归一化、CLAHE、热红外 TIFF 反转、Otsu、局部增强、去畸变和必要的图像尺度归一。该层的职责是为标定、匹配和重建提供稳定输入。

### 4.4 matching

负责 TWMM 适配、输入输出转换、对应点管理、异常值剔除和单应性估计。生产实现必须优先使用 TWMM-main/TAMM_clean，不重新实现论文核心逻辑。

### 4.5 radiometry

负责热红外 TIFF 主输入帧、DJI SDK 元数据、ExifTool 信息和环境参数的整合，输出标准化的温度矩阵。该层必须区分原始传感器值、辐射校正中间值和最终温度值。

### 4.6 metashape

负责 Metashape 项目自动化，包括导入、对齐、优化、深度图构建、稠密点云生成和导出。该层必须与通用算法层隔离，避免把摄影测量自动化逻辑散落到其他模块。

### 4.7 geometry

负责重投影、遮挡判断、坐标变换、插值和可见性判定。该层只输出几何证据，不直接承担温度赋值。

### 4.8 enrichment

负责将热信息分配到三维点，执行可见性过滤、重投影误差过滤、权重融合和缺失值处理。该层消费 geometry、matching 和 radiometry 的输出。

### 4.9 io

负责图像、XML、JSON、CSV、PLY / LAS、日志和清单文件的读写。io 层必须稳定、可审计，并尽量避免隐式格式转换。日志输出应通过统一的入口配置，避免各业务模块重复初始化根日志器。

### 4.10 validation

负责单元测试、集成测试、质量评估、可视化检查材料和回归样例管理。validation 不生产业务结果，只验证其他层的结果是否满足门槛。它也消费运行日志作为失败回放与审计依据。

## 5. 运行模式

- 标定模式：只处理标定数据，输出标定结果和诊断图。
- 匹配模式：只处理去畸变后的双光谱图像对，输出对应点和单应性。
- 辐射模式：只处理热红外 TIFF 主输入帧与元数据，输出温度矩阵。
- 重建模式：只处理 RGB 图像与 Metashape 项目，输出稠密点云和相机位姿。
- 富集模式：消费重投影记录、单应性和温度矩阵，输出带温度属性的点云。
- 批处理模式：按数据集清单顺序执行上述多个模式，并记录每个阶段的状态。

## 6. 跨层约束

- 只有 matching 层和 enrichment 层允许热红外与可见光发生直接交互。
- metashape 层只负责几何重建，不承担辐射校正逻辑。
- geometry 层不直接修改温度，只提供点位关系和可见性证据。
- radiometry 层不依赖点云，不关心三维几何。
- 任何输出都必须带有版本、来源和单位信息。
- 任何关键阶段都应有与输出目录绑定的运行日志，且日志内容必须能回放阶段状态变化、异常和摘要信息。
- dataset_profile 层记录的物理事实是单一来源，其他文档只能引用，不得重新发明一套不同的分辨率、尺寸或标定板几何。
- runtime_config 层记录的运行参数是单一来源，其他文档只能引用，不得把同一组路径、阈值或开关分散写成多个版本。

## 7. 质量门槛

- 标定阶段：棋盘格识别率、重投影误差、参数合理性。
- 匹配阶段：对应点数量、异常值比例、单应性稳定性、叠加效果。
- 辐射阶段：温度范围、单位一致性、环境参数是否生效。
- 重建阶段：点云密度、重投影误差、导出成功率。
- 富集阶段：温度覆盖率、插值合理性、融合一致性和异常值分布。

## 8. 版本与审计

- 每个阶段的结果都应带 schema_version 和 source_version。
- 运行清单必须显式引用 dataset_profile 和 runtime_config 的版本或摘要哈希。
- 原始数据与派生数据应分目录保存，禁止覆盖原始输入。
- 关键输出必须能够追溯到输入数据和参数版本。
- 任何一次算法改动都应伴随相应设计文档和质检材料更新。

## 9. 非目标

- 不是通用摄影测量平台。
- 不是通用图像配准库。
- 不是对 TWMM 的重新发明。
- 不是把 Metashape 替换成自研 SfM / MVS。
